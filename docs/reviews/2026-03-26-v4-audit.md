# V4 Audit Against Current ToS

Date: 2026-03-26

## Scope

This audit reads `tos_kag_project_scheme_v4_master_1_12.xlsx` as the current
KAG-facing intake schema for Zarathustra Prologue 1 and compares it against the
current `Tree-of-Sophia` repository state after the canonical `sources/`,
`intake/`, and `tree/` landing.

The goal is not to treat the workbook as ToS authority.
The goal is to use `v4` as an audit frame for what the next raw intake pack
should contain and what is already present, partial, or absent inside ToS.

## What V4 Adds

Relative to the older start-schema workbook, `v4` keeps the same core table
family and adds four coordination surfaces:

- `13_Proektnaya_skhema` as one working matrix of waves, required fields,
  anti-patterns, and spawn triggers
- `14_Poryadok_prokhoda` as the intended traversal order from corpus address to
  late projections
- `15_Master_svyaznost_1_12` as one master connectivity table for paragraphs
  `[1]-[12]`
- `16_Master_pokrytie_1_12` as one audit sheet for coverage and row
  distribution across the master table

The workbook's own project logic is:

- fundamental wave: `corpus_map`, `witnesses`, `segments`, `nodes`,
  `event_state_nodes`, `edges`, `translation_tensions`
- second wave: `witness_glosses`, `principles`
- third wave: `analogy_nodes`, `reasoning_paths`, `lineage_links`,
  `agent_witnesses`, `query_views`

The flow sheet says the current route should move in this order:

1. `corpus_map`
2. `witnesses`
3. `segments`
4. `nodes`
5. `event_state_nodes`
6. `edges`
7. `translation_tensions`
8. `witness_glosses`
9. `principles`

`v4` therefore expects the first real raw pack for Prologue 1 to be
graph-bearing, not just source-bearing.

  `intake/thus-spoke-zarathustra/prologue-1/mode-b/README.md`

- `3` witness layers
- `12` shared paragraph-aligned segments

But these are still embedded in one canonical source-node surface rather than
exposed as first-class `v4` raw tables.

## Concrete Reading

The practical result is:

- ToS is already structurally ready for a `v4`-aligned raw intake pack.
- The first true gap is not source material anymore.
- The first true gap is graph-bearing extraction.

In other words:

- source-facing groundwork is already present
- canonical bounded route is already present
- intake path is already present
- the missing layer is the candidate tabular graph field expected by `v4`

## What The First Raw Pack Should Contain

For Prologue 1, the first useful raw intake pack should include at least:

1. `corpus_map`
2. `witnesses`
3. `segments`
4. `nodes`
5. `event_state_nodes`
6. `edges`
7. `translation_tensions`

`witness_glosses` and `principles` are valuable, but they are not blockers for
the first landing if the goal is to open the graph-bearing candidate layer
cleanly.

The third-wave tables in `v4` should remain deferred.

## Audit Verdict

`v4` is not blocked by current ToS architecture.

The repository is ready to receive a first `v4`-aligned raw pack under:

- `intake/thus-spoke-zarathustra/prologue-1/mode-b/`

The next honest move is therefore:

- bring in the raw candidate tables for Prologue 1
- audit them against the `v4` master coverage targets
- then decide what is promoted from `intake/` into canonical `tree/`

## Risks To Watch

- Do not let the workbook become a second hidden canon over ToS.
- Do not collapse embedded canonical node data and raw candidate tables into one
  authority layer.
- Do not jump into third-wave surfaces before the fundamental tables exist.
- Do not treat `15_Master_svyaznost_1_12` as canonical authored truth; it is
  still a candidate control surface until reviewed.

## Bottom Line

Current ToS is ready for the first `v4` raw graph pack.

What is still missing is not doctrine, not source, and not route scaffolding.
What is missing is the first explicit candidate table set for:

- `corpus_map`
- `witnesses`
- `segments`
- `nodes`
- `event_state_nodes`
- `edges`
- `translation_tensions`
