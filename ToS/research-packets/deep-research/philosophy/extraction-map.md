# Philosophy Packet Extraction Map

## Role

This map names how future research packets can move toward tree and graph
surfaces without becoming those surfaces.

## Route

```text
research packet
  -> branch review
    -> local branch skeleton
      -> proposed nodes
        -> proposed relations
          -> relation pack
            -> canon promotion
              -> derived graph/export
```

## Translation Points

| Packet pressure | Review question | Future route |
| --- | --- | --- |
| table row | What branch, period, and evidence posture does this suggest? | `ToS/philosophy/` branch review |
| dossier section | Which object kind is being proposed? | local branch workbench |
| source lead | What real witness, edition, translation, or publication anchors it? | `ToS/source-witnesses/` or branch `sources/` |
| proposed object | Is this a corpus, work, figure, institution, medium, concept, controversy, or transmission? | `graph-workbench/proposed-nodes/` |
| proposed connection | Which relation kind and graph layer does it need? | `graph-workbench/proposed-relations/` |
| reviewed relation set | Are endpoints and predicates ready for canon review? | relation pack |
| promoted structure | Which generated view can read it without owning it? | `ToS/derived-exports/` |

## Object-Kind Handoff

Research packets may suggest objects. The owning branch decides whether the
object becomes a branch-local scaffold, a proposed node, a source witness, a
canon candidate, or a discarded lead.

No packet field should skip branch review and write directly into canon or
derived exports.
